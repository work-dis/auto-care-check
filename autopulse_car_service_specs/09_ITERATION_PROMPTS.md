# 09. Поэтапные промпты для нейросети

Используйте их после того, как передадите ИИ базовые файлы ТЗ. Каждый следующий промпт отправляйте только после того, как предыдущая итерация собрана и проверена.

## Промпт 1 — аудит и фундамент

```text
Изучи проект и приложенное ТЗ AutoPulse. Не начинай с визуальной вёрстки. Сначала покажи краткий аудит: текущий стек, структура, что уже есть, что нужно добавить. Затем создай IMPLEMENTATION_PLAN.md по итерациям из ТЗ.

Реализуй фундамент: строгий TypeScript, конфигурацию окружения, PostgreSQL + Prisma, миграции, seed data, Docker Compose, healthcheck и базовую auth/demo-auth архитектуру. Не сканируй node_modules/.next/dist/coverage без необходимости. Добавь README с командами запуска.

После работы сообщи изменённые файлы и фактически запущенные проверки.
```

## Промпт 2 — автомобиль, пробег, планы обслуживания

```text
Продолжай AutoPulse по ТЗ. Реализуй модели и API/UI для автомобиля, текущего пробега, истории пробега, категорий и MaintenancePlan.

Обязательные правила:
- поддержка сроков date_only, mileage_only, whichever_comes_first и manual;
- архивирование вместо удаления для сущностей с историей;
- уменьшение пробега возможно только как correction с обязательной причиной;
- Zod-валидация форм и API;
- backend ownership checks.

Сделай удобные формы для desktop и mobile, но без финального dashboard. Добавь тесты на валидацию и ownership. В конце покажи, что было реализовано и какие тесты реально запускались.
```

## Промпт 3 — status engine и dashboard-приборная панель

```text
Продолжай AutoPulse. Реализуй domain/service layer status engine по файлу 03_DATA_MODEL_AND_BUSINESS_LOGIC.md. Он обязан возвращать status, remainingDays, remainingMileage, nextDueAt, nextDueMileage и понятную statusReason.

Покрой тестами: date_only, mileage_only, whichever_comes_first, overdue, soon, watch, normal, unknown и пересчёт после нового одометра.

Затем сделай /dashboard как современную цифровую автомобильную панель: графитовый фон, крупный пробег, блок «Готовность по обслуживанию», «Сделать сейчас», «Скоро понадобится», «Под присмотром», расходы и последняя работа. Не имитируй конкретный автомобильный бренд, не называй score диагностикой и всегда показывай дисклеймер.

UI должен быть полностью responsive: sidebar на desktop и нижняя навигация на mobile.
```

## Промпт 4 — сервисная история, расходы и завершение работ

```text
Продолжай AutoPulse. Реализуй ServiceRecord, ServiceRecordPlanItem и ServicePart. Пользователь должен отметить одну или несколько работ выполненными, указать дату, пробег, сервис, стоимость работ, запчасти, фото чека и заметки.

При подтверждении записи транзакционно:
1) создай snapshots истории;
2) обнови одометр;
3) обнови lastCompletedAt/lastCompletedMileage плана;
4) пересчитай следующий срок;
5) подготовь отмену старых напоминаний;
6) сохрани audit event.

Реализуй историю и расходы по категориям. Подтверждённые записи не удаляй физически: разреши void с причиной и корректный пересчёт. Добавь тесты на этот flow.
```

## Промпт 5 — напоминания и центр уведомлений

```text
Продолжай AutoPulse. Реализуй ReminderRule и in-app Notification Center. Поддержи быстрые правила: 30/14/7/1 дней, 3000/1000/500 км, день срока, повтор после просрочки и точную дату/время.

Сделай отдельный worker/cron, который создаёт уведомления идемпотентно через dedupeKey. Не отправляй уведомления в пользовательском HTTP-запросе. После подтверждённой сервисной работы напоминания старого цикла должны стать stale/cancelled.

Добавь UI настройки времени, тихих часов и отключения одного правила. Добавь тесты на дедупликацию и отмену устаревшего уведомления.
```

## Промпт 6 — наблюдения, polish и финальная проверка

```text
Заверши AutoPulse по ТЗ. Добавь раздел «Нужно проверить» / observations: создать заметку о симптоме, выбрать приоритет, фото, напоминание, статус open/watching/service_planned/closed и связать с сервисной записью.

Дополни empty/loading/error/success states, доступность, responsive UI, аккуратные анимации reduced-motion-safe, PWA metadata и финальную документацию.

Запусти и честно зафиксируй: lint, typecheck, tests, production build. Пройди demo-flow от создания автомобиля до переноса следующего срока после выполненной работы. Список известных ограничений оставь в README.
```
