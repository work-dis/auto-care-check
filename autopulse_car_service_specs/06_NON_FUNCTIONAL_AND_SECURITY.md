# 06. Нефункциональные требования, безопасность и качество

## 1. Производительность

- Первый meaningful render дашборда должен быть быстрым на среднем мобильном устройстве и медленном LTE.
- Не загружать все фотографии чеков и всю историю работ на стартовом экране.
- Дашборд должен получать компактный агрегированный ответ, а не выполнять N+1 запросов на каждую карточку.
- Списки истории, уведомлений и наблюдений — с курсорной пагинацией.
- Пересчёт статусов должен быть безопасным для десятков планов на автомобиле и не зависеть от браузера пользователя.

## 2. Надёжность данных

- PostgreSQL — источник истины.
- Все изменения, которые могут повлиять на историю, выполняются в транзакции.
- Подтверждение сервисной записи, обновление плана и отмена старых напоминаний — одна согласованная операция.
- Обязательны миграции БД и возможность поднять пустую среду с нуля.
- Регулярный резервный backup базы и проверяемая процедура восстановления.

## 3. Безопасность

- Строгая серверная проверка владельца каждой сущности.
- Пароли не хранятся в открытом виде.
- Сессии/токены защищены: secure cookies, ограничение срока жизни, защита от CSRF в выбранной модели auth.
- Rate limit на auth и чувствительные mutation endpoints.
- Валидация файлов: тип, размер, антивирусная/безопасная обработка при доступности; не доверять расширению файла.
- Приватные фото чеков не должны иметь предсказуемые публичные URL без контроля доступа.
- VIN и госномер считать чувствительными для UX: маскировать в обычных списках, не логировать целиком.
- Не писать секреты, токены, пароли и персональные данные в консольные логи.

## 4. Конфиденциальность

- Пользователь должен иметь возможность выгрузить свои данные в понятном формате.
- Удаление аккаунта должно иметь чёткий flow: предупредить о последствиях, сделать grace period или безопасное удаление по политике продукта.
- Вложения и резервные копии должны учитываться в политике удаления.
- Не передавать автомобильные данные третьим сторонам без явного согласия.

## 5. Доступность

- Контраст уровня WCAG AA.
- Не использовать цвет как единственный носитель значения.
- Корректная навигация с клавиатуры на desktop.
- Логичные aria-label для кнопок-иконок.
- Поддержка reduced motion.
- Формы дают текстовые ошибки рядом с полем и сохраняют уже введённые данные.

## 6. Локализация и форматы

- UI на русском в MVP.
- Строки не хардкодить внутри компонентов: заложить структуру i18n для будущих EN/PL.
- Даты и время показывать в timezone пользователя.
- Валюты/суммы форматировать через Intl API.
- Пробег в MVP — км, но модель должна допускать мили без миграции архитектуры.

## 7. Логи, мониторинг и эксплуатация

- Structured logs c requestId/correlationId.
- Логировать не секреты, а технические события: ошибка отправки уведомления, ошибка миграции, недоступность базы.
- Healthcheck для API, web и worker.
- Docker Compose: отдельные сервисы `web`, `api`/`server`, `worker`, `postgres`.
- `.env.example` без реальных секретов.
- CI: lint, typecheck, unit tests, build, migration check.

## 8. Качество кода

- TypeScript strict mode.
- Нет `any` без документированной причины.
- Единые типы DTO/схемы на уровне API.
- Компоненты маленькие и композиционные; не писать один огромный `Dashboard.tsx` на тысячу строк.
- Бизнес-правила покрываются unit-тестами.
- Не скрывать ошибки форм только toast-уведомлением; пользователь должен понимать, что исправить.

## 9. Ограничение медицинских/безопасностных заявлений

Приложение не ставит технический диагноз. Даже при красных карточках тексты должны быть вроде:

- «По вашему плану срок обслуживания наступил»;
- «Проверьте узел в сервисе, если симптом сохраняется».

Не использовать утверждения «ездить опасно», «двигатель неисправен» или «тормоза сломаны» без диагностических данных.
