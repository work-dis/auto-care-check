# 07. Roadmap разработки и критерии приёмки

## Общий принцип

Каждая итерация должна закончиться рабочим, проверяемым результатом. Не переходить к «красивой анимации» до того, как корректно работают история, статусы, доступы и напоминания.

## Итерация 0 — фундамент проекта

### Результат

- Репозиторий структурирован.
- TypeScript strict, lint, format, env validation.
- Docker Compose с PostgreSQL.
- Prisma schema и миграции.
- Базовая авторизация или безопасный demo-auth для локальной разработки.
- Seed-данные с одним автомобилем и несколькими задачами обслуживания.

### Acceptance criteria

- Новый разработчик поднимает проект по README.
- `docker compose up` запускает нужные сервисы.
- Миграции применяются без ручного редактирования БД.
- Есть healthcheck endpoint.

## Итерация 1 — автомобили, пробег, каталог планов

### Результат

- CRUD автомобиля.
- Ввод текущего пробега и история одометра.
- CRUD категорий и планов обслуживания.
- Архивирование/восстановление вместо физического удаления.

### Acceptance criteria

- Пользователь не может видеть чужой автомобиль.
- Пробег меньше последнего требует явной причины корректировки.
- Архивный план не исчезает из истории.
- Можно создать план с датой, пробегом и правилом «что наступит раньше».

## Итерация 2 — status engine и dashboard

### Результат

- Backend/domain service рассчитывает дату, пробег, статус и текстовое объяснение.
- Endpoint дашборда возвращает агрегированный ответ.
- Интерфейс «цифрового табло» с главными блоками состояния.

### Acceptance criteria

- Изменение пробега обновляет статус задач по километрам.
- Просроченная по дате задача видна в блоке «Сделать сейчас».
- Для недостаточных данных показывается объяснение, что именно нужно заполнить.
- Индекс готовности не показан как «диагноз автомобиля».

## Итерация 3 — сервисная история и расходы

### Результат

- Форма добавления работ, запчастей, стоимости, чека.
- Связь записи с планами обслуживания.
- Автоматическое обновление последнего обслуживания и следующего срока.
- История и базовая аналитика расходов.

### Acceptance criteria

- После «Замена масла выполнена» задача переходит в normal с новым сроком.
- Старая цена и название остаются в исторической записи после изменения плана.
- Отмена записи не удаляет её без следа и корректно пересчитывает связанные планы.

## Итерация 4 — напоминания

### Результат

- ReminderRule UI с готовыми вариантами и custom values.
- In-app notification center.
- Worker/cron с идемпотентным созданием уведомлений.
- Отмена устаревших уведомлений после сервисной записи.

### Acceptance criteria

- Нельзя получить два одинаковых напоминания на один и тот же триггер.
- Уведомление по пробегу появляется после обновления одометра, когда достигнут порог.
- После отметки обслуживания старое напоминание не приходит.
- Пользователь может выключить одно правило, не отключая весь план.

## Итерация 5 — наблюдения, polish, PWA

### Результат

- «Нужно проверить» / observations.
- Улучшение мобильных форм и empty states.
- PWA metadata, offline-friendly shell при необходимости.
- Тесты, доступность, финальная проверка дизайн-системы.

### Acceptance criteria

- Наблюдение можно закрыть либо связать с сервисной работой.
- В мобильной версии все основные действия доступны без горизонтального скролла.
- Lighthouse/ручная проверка не показывает очевидных проблем с контрастом и touch targets.

## Definition of Done для каждой задачи

Задача считается законченной, если:

1. Реализована функциональность и типы.
2. Добавлены/обновлены unit или integration tests для критичной логики.
3. Ошибки обработаны и понятны пользователю.
4. UI адаптивен для desktop и mobile.
5. Нет физического удаления исторически связанных сущностей.
6. Изменения документированы в README/ADR при необходимости.
7. `lint`, `typecheck`, `test`, `build` проходят.

## Критерии первой демо-версии

Демо готово, когда пользователь может:

1. Создать автомобиль.
2. Записать текущий пробег.
3. Добавить «Замена масла» с периодом 10 000 км или 12 месяцев.
4. Увидеть, что делать сейчас и что скоро.
5. Настроить напоминание за 1 000 км и за 30 дней.
6. Отметить замену масла выполненной с суммой и фото чека.
7. Увидеть новую дату/пробег следующей замены и сохранённую историю.
