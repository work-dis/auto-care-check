# 02. Функциональные требования

## 1. Авторизация и профиль

### Обязательное

- Регистрация через e-mail и пароль либо magic link; архитектура должна позволять добавить OAuth позже.
- Пользователь может изменить имя, часовой пояс, язык интерфейса и настройки уведомлений.
- По умолчанию часовой пояс берётся из браузера, но пользователь может его изменить.
- Нельзя показывать данные одного пользователя другому без явного механизма доступа.

### Настройки уведомлений

- Включить/выключить каналы: in-app, e-mail, push/Telegram — только если канал подключён.
- Выбрать время по умолчанию для напоминаний, например 09:00.
- Задать quiet hours, когда несрочные уведомления не отправляются.
- Настроить частоту повторов просроченных задач.

## 2. Автомобили

Пользователь может создавать, редактировать, архивировать и переключать несколько автомобилей.

### Поля автомобиля

| Поле | Обязательное | Примечание |
|---|---:|---|
| Название в приложении | Да | Например, «Passat B3» или «Семейная машина» |
| Марка | Да | Текст/справочник |
| Модель | Да | Текст/справочник |
| Год выпуска | Нет | Четыре цифры |
| Текущий пробег | Да | Неотрицательное число в км |
| Единица пробега | Да | В MVP — км, структура готова для mi |
| Госномер | Нет | Не показывать публично |
| VIN | Нет | Маскировать в UI, показывать по действию |
| Топливо | Нет | Бензин/дизель/гибрид/электро/другое |
| Коробка | Нет | МКПП/АКПП/робот/вариатор/другое |
| Двигатель | Нет | Свободный текст |
| Фото | Нет | Обложка автомобиля |
| Заметка | Нет | Особенности, например «масло 5W-40» |

### Правила

- Архивированный автомобиль не удаляется вместе с историей.
- При архивировании автомобиля пользователь видит предупреждение, что записи сохранятся, но новые уведомления по нему перестанут создаваться.
- Основной автомобиль можно выбрать отдельно; он открывается при входе в приложение.

## 3. Одометр и пробег

### Быстрое обновление

- На дашборде должна быть заметная кнопка «Обновить пробег».
- Поле принимает значение текущего одометра и, опционально, дату измерения.
- После сохранения система пересчитывает статусы и напоминания, связанные с пробегом.
- Если пользователь ввёл пробег меньше предыдущего подтверждённого значения, система должна показать предупреждение и запросить причину корректировки.

### История пробега

- Каждое изменение фиксируется как `OdometerReading` с датой, значением, источником (`manual`, `service_record`, `import`) и комментарием.
- При сохранении сервисной работы пробег этой работы также создаёт/обновляет запись в истории пробега.
- Пользователь может видеть график/список последних показаний.

## 4. План обслуживания

### Категории по умолчанию

- Двигатель и масло.
- Фильтры.
- Тормозная система.
- Шины и колёса.
- Подвеска и рулевое.
- Жидкости.
- Аккумулятор и электрика.
- Кондиционер.
- Кузов/сезонные работы.
- Документы и обязательные проверки.
- Пользовательские пункты.

### Карточка плана обслуживания

| Поле | Описание |
|---|---|
| Название | Например, «Замена моторного масла» |
| Категория | Одна из системных или пользовательская |
| Описание | Что именно контролируется/меняется |
| Тип | Плановая работа / Осмотр / Наблюдение / Документ |
| Интервал по времени | Например, 12 месяцев |
| Интервал по пробегу | Например, 10 000 км |
| Логика срока | Раньше из двух / только дата / только пробег |
| Последнее обслуживание | Дата и пробег последнего выполнения |
| Следующий срок | Рассчитывается или задаётся вручную |
| Порог «Скоро» | Дни и/или км до срока |
| Порог «Под присмотром» | Более раннее предупреждение |
| Приоритет | Обычный / высокий / критичный вручную |
| Активность | Активен / архив / отключён |
| Заметки | Свободный текст |

### Готовые шаблоны

В MVP дать минимум один редактируемый шаблон «Базовое обслуживание» с типовыми пунктами. Он не должен обещать точные регламенты конкретной марки автомобиля. У каждого пункта есть подпись: «Проверьте интервал по руководству именно вашего автомобиля».

## 5. Статусы и карточки состояния

Каждый активный пункт плана должен иметь один статус:

- **Просрочено** — наступил срок по дате или пробегу.
- **Скоро** — до срока осталось меньше установленного порога.
- **Под присмотром** — срок ещё не близок, но есть ручная отметка, симптом либо ранний предупредительный порог.
- **В норме** — данных достаточно, срок не близок.
- **Нет данных** — невозможно рассчитать срок: нет последнего обслуживания, интервала или показания пробега.
- **Отключено** — пункт временно исключён из расчётов.

Каждая карточка обязана объяснять причину статуса текстом: например, «Осталось 12 дней», «Просрочено на 430 км», «Нужна дата последней замены», «Пользователь отметил посторонний звук 2 дня назад».

## 6. История работ и расходов

### Создание работы

При создании сервисной записи пользователь выбирает автомобиль и один либо несколько пунктов плана. Поля:

- дата работы;
- пробег на дату работы;
- место/сервис/мастер;
- выполненные операции;
- запчасти и расходники: название, артикул опционально, количество, цена;
- стоимость работы;
- общая сумма;
- валюта;
- ссылка/фото чека;
- заметка;
- флаг «подтверждённая запись».

### Результат сохранения

- Создаётся неизменяемый исторический снимок названия работ и цен на момент записи.
- Соответствующий план обслуживания обновляет `lastCompletedAt` и `lastCompletedMileage`.
- Следующий срок пересчитывается по интервалу.
- Невыполненные уведомления, которые относились к закрытому циклу обслуживания, отменяются/помечаются устаревшими.

### Правило «архивировать, а не удалять»

- Пользователь не должен физически удалять категорию/план, если по нему есть история.
- Вместо удаления есть архивирование: пункт перестаёт показываться среди активных, но остаётся в истории и может быть восстановлен.
- Если запись работы была создана ошибочно, её можно отменить/архивировать с причиной. Полное необратимое удаление — только для пустого черновика без зависимостей.

## 7. Напоминания

### Варианты триггеров

Пользователь может поставить одно или несколько правил на один пункт:

- за N дней до даты;
- в день наступления срока;
- за N км до срока;
- при наступлении срока по пробегу;
- повторно каждые N дней после просрочки;
- в конкретную дату и время;
- после ручной заметки: «напомнить проверить завтра/через неделю».

### UX создания напоминания

В форме должны быть готовые быстрые кнопки:

- За 30 дней, 14 дней, 7 дней, 1 день.
- За 3 000 км, 1 000 км, 500 км.
- В день срока.
- Каждые 7 дней после просрочки.
- Своя дата/время.

Пользователь выбирает канал, время, повтор и может отключить одно конкретное правило, не выключая весь план обслуживания.

## 8. Дашборд

Главная страница отображает выбранный автомобиль и состоит из:

1. верхней полосы со сменой автомобиля, текущим пробегом и кнопкой обновления пробега;
2. большого блока «Состояние обслуживания» с понятным индексом готовности и расшифровкой;
3. блока «Нужно сделать сейчас»;
4. блока «Скоро понадобится»;
5. блока «Под присмотром»;
6. горизонтальной/вертикальной ленты ближайших действий;
7. кратких расходов за 30 дней и с начала года;
8. последней сервисной записи;
9. быстрых действий: добавить работу, добавить заметку, обновить пробег, настроить напоминание.

## 9. История и аналитика

- Список работ с фильтрами по периоду, категории, автомобилю, сервису и статусу.
- Просмотр одной записи со всеми деталями и вложениями.
- Сумма расходов за период.
- Расходы по категориям.
- Средняя стоимость обслуживания за месяц/год — только если данных достаточно.
- Экспорт в CSV/JSON как отдельная задача после базового MVP.

## 10. Наблюдения и проблемы

Пользователь может создать не только плановую работу, но и «наблюдение»:

- название;
- описание симптома;
- дата создания;
- приоритет;
- фото;
- связанная категория;
- напоминание;
- статусы: открыто / наблюдаю / передано в сервис / закрыто.

Эта сущность не должна автоматически объявлять неисправность. В интерфейсе использовать нейтральную формулировку «Нужно проверить».
