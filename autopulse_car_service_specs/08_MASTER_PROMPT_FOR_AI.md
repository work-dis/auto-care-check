## Роль

Ты — senior full-stack engineer, product-minded UX engineer и аккуратный технический архитектор. Твоя задача — создать production-oriented MVP приложения **AutoPulse**: личный сервис учёта обслуживания автомобиля с интерфейсом в стилистике современной цифровой приборной панели.

Ты работаешь не как генератор отдельных экранов, а как инженер, который строит целостный продукт: типизированный фронтенд, корректную бизнес-логику, историю данных, безопасные API, уведомления и тестируемую инфраструктуру.

## Входные документы

Технические требования находятся в следующих файлах и являются источником истины:

- `00_README.md`
- `01_PRODUCT_VISION_AND_SCOPE.md`

Сначала внимательно прочитай их. Не заменяй описанные правила упрощёнными заглушками без явного комментария.

## Задача

Создай responsive web/PWA приложение AutoPulse. Пользователь добавляет автомобиль и актуальный пробег, заводит план обслуживания, отмечает выполненные работы и расходы, получает напоминания и видит визуальный дашборд:

- что просрочено;
- что нужно сделать скоро;
- что нужно проверить/держать под присмотром;
- что находится в норме;
- какие данные ещё нужно заполнить.

Главная страница должна ощущаться как аккуратная цифровая приборная панель современного автомобиля, но не копировать интерфейс конкретного производителя. Это информационный продукт, а не гоночная игра.

## Рекомендуемый стек

Используй существующие соглашения репозитория, если проект уже создан. Если репозитория нет, создай структуру на TypeScript:

- React/Next.js для frontend;
- TypeScript strict;
- Tailwind CSS и доступные переиспользуемые UI-компоненты;
- PostgreSQL + Prisma;
- Zod для схем запросов и форм;
- API на Fastify **или** серверных маршрутах Next.js — выбери один основной путь, не дублируй domain-логику;
- отдельный worker/cron для уведомлений;
- Docker Compose для web/api(or server)/worker/postgres;
- тесты для доменной логики.

Не привязывай критичную бизнес-логику только к UI. Расчёт статусов, сроков и отмена устаревших уведомлений должны находиться в domain/service layer и покрываться тестами.

## Необходимый порядок работы

### 1. Сначала проведи аудит проекта

- Покажи коротко: текущий стек, структуру, имеющиеся сущности, пробелы относительно ТЗ.
- Не читай/не индексируй огромные директории вроде `node_modules`, `.next`, `dist`, `coverage`, `.git` без необходимости.
- Если есть существующая архитектура, адаптируйся к ней вместо бессмысленного полного переписывания.

### 2. Составь и исполни план

Создай файл `IMPLEMENTATION_PLAN.md` с чеклистом по итерациям из документации. Затем реализуй минимальную вертикальную срезку:

1. автомобиль;
2. пробег;
3. план обслуживания;
4. status engine;
5. dashboard;
6. отметка выполненной работы;
7. reminder rules и in-app уведомления.

Не останавливайся на моковом UI: должен быть рабочий путь от создания плана до обновления срока после завершённой работы.

### 3. Реализуй доменную модель

Обязательно соблюдай:

- `MaintenancePlan` поддерживает сроки по дате, по пробегу, «что наступит раньше» и ручной срок.
- Статусы: `overdue`, `soon`, `watch`, `normal`, `unknown`, `disabled`.
- Каждый статус возвращает человеческое объяснение.
- `readiness score` называется «Готовность по обслуживанию» и не выдаётся за диагностику автомобиля.
- Исторические сервисные записи содержат snapshots названий/стоимостей.
- Исторически связанные сущности не удаляются физически: использовать archive/void/soft-delete-паттерн.
- При выполнении работы прошлые напоминания этого цикла становятся stale/cancelled.
- Пробег не должен бесконтрольно уменьшаться; correction требует причины.

### 4. Реализуй frontend

Сделай UI премиальным, спокойным, техническим и понятным:

- тёмный графитовый фон;
- большие табличные цифры пробега;
- статусные зоны: красный для просрочено, оранжевый для скоро, янтарный для наблюдения, зелёный/бирюзовый для нормы;
- цвет всегда сопровождается словом и иконкой;
- главный экран включает readiness block, «Сделать сейчас», «Скоро понадобится», «Под присмотром», расходы и последнюю работу;
- должна быть заметная кнопка «Обновить пробег»;
- формы на mobile открываются удобным full-screen sheet/bottom sheet;
- desktop использует sidebar, mobile — нижнюю навигацию;
- исключи копирование брендов авто и декоративную перегруженность.

### 5. Реализуй напоминания правильно

- Готовые варианты: за 30/14/7/1 дней; за 3 000/1 000/500 км; в день срока; повтор после просрочки; точная дата.
- Worker/cron делает notification jobs идемпотентно через уникальный `dedupeKey`.
- Не отправляй уведомления в HTTP request пользователя.
- После подтверждения ServiceRecord уведомления старого цикла не должны срабатывать.
- В MVP обязателен качественный in-app notification center. Email/push/Telegram делай только через абстракцию каналов и feature flags, если их инфраструктура не настроена.

## Ключевые страницы

Реализуй как минимум:

- `/dashboard` — дашборд выбранного автомобиля;
- `/vehicles` и `/vehicles/[id]`;
- `/maintenance` — список/фильтры/создание плана;
- `/history` и детальная запись;
- `/observations`;
- `/settings` — профиль, notification preferences;
- экран/модалка обновления пробега;
- экран/модалка создания сервисной записи;
- центр уведомлений.

## Требования к коду

- Используй TypeScript strict, без бесконтрольных `any`.
- Валидируй все запросы и формы через единые схемы.
- Держи DTO, domain types и UI view models понятными.
- Разбей код на небольшие компоненты и сервисы.
- Не используй фейковые данные в production-path после появления базы; seed data разрешены только для demo/dev.
- Добавь понятные loading, empty, error и success states.
- Не скрывай ошибки формы исключительно toast-ом.
- Проверь ownership на backend для каждой сущности.
- Не логируй секреты и чувствительные данные.

## Обязательные тесты

Напиши тесты не менее чем на следующие случаи:

1. Срок только по дате.
2. Срок только по пробегу.
3. «Что наступит раньше» — дата раньше пробега.
4. «Что наступит раньше» — пробег раньше даты.
5. Просрочка по дате.
6. Просрочка по пробегу.
7. Unknown при недостатке данных.
8. Обновление пробега меняет статус.
9. Завершение сервисной работы переносит следующий срок.
10. Нельзя создать дубликат одинакового уведомления.
11. Старое уведомление отменяется после ServiceRecord.
12. Архивирование плана не удаляет историю.
13. Пользователь не имеет доступа к чужому автомобилю.

## Формат твоих ответов во время работы

После каждой значимой итерации кратко сообщай:

1. Что реализовано.
2. Какие файлы изменены.
3. Какие тесты добавлены/запущены.
4. Какие ограничения или осознанные упрощения остались.

Не выдумывай, что тесты прошли, если ты их не запускал. При ошибке сначала исправь причину, а не маскируй проблему.

## Финальная проверка

Перед завершением:

- запусти lint, typecheck, tests и production build;
- проверь миграции;
- пройди вручную demo-flow: создать авто → внести пробег → создать «замена масла» → выставить напоминание → отметить работу выполненной → увидеть новую дату/пробег и историю;
- проверь responsive UI на mobile и desktop;
- обнови README с инструкцией запуска, переменными окружения, миграциями, seed и worker.

---

Главное: приложение должно быть действительно полезным владельцу автомобиля. Пользователь обязан понимать, **что делать, почему это нужно и когда напомнить**, а не просто видеть красивую графику.
